The Big Picture

Two origination engines, one analysis pipeline

Founders' Oak runs deal flow like a search fund with an automation layer bolted on. Opportunities enter through two channels — deals that come to us (brokered) and deals we go find (proprietary). Both funnel into the same scoring, modeling, research, and offer machinery.

Channel 1 · Inbound

Brokered

BizBuySell scraping + inbound broker relationships. Listings come pre-packaged with a CIM. High volume, more competition.

Channel 2 · Outbound

Proprietary

Thesis-driven, off-market origination. We build verified owner lists by industry + geography and reach out directly. Lower volume, no auction.

Channel 1 · Inbound

Brokered deal flow

Deals that come to us: business-for-sale listings scraped from BizBuySell, plus inbound broker outreach (direct emails, teasers, off-platform introductions). Higher volume and pre-packaged with a CIM, but competitive — often an auction. The job here is to filter fast, understand the listing, and get behind the NDA before running full analysis.

Filter: Cash flow ≥ $750K Priority states: TX, OK, AR Classify: 17-category taxonomy ID format: numeric (BizBuySell) · INB-xxx (inbound)
1Source & scrape BizBuySell (multi-actor failover)Automated

Scrape new BizBuySell listings above the cash-flow floor, dedupe against everything already in the tracker, and stage the fresh ones. Rather than depending on a single Apify actor, the scraper runs a tiered failover: it tries each enabled actor in priority order until one returns listings, auto-widens the date window (3d → 7d → 30d) if a run comes back thin, dedupes by BizBuySell productId, and stops loudly (an alert email) only if every actor returns zero. This is the top of the inbound funnel.

Runs: bizbuysell-daily-digest — Mon/Wed/Fri 5:00pm · direct Apify API (not the MCP), item cap 200/actor
Tools
scrape_bizbuysell.py crawlerbros — primary, enriched fatihtahta — failover good_cheap — failover acquistion-automation / lexis — paid, off dedupe by productId
Inputs
  • BizBuySell live listings
  • Cash-flow filter ≥ $750K
  • Existing tracker rows (dedupe key)
  • ACTOR_TIERS priority list
Outputs
  • New listing records (enriched: CF, gross rev, EBITDA)
  • Deduped candidate set
  • Loud all_actors_empty alarm if all fail
Why multi-actor: on 2026-07-03 the long-time production actor (fatihtahta) died — a BizBuySell-side change zeroed it out on every build and input. A single-actor scraper would have silently returned 0 listings. The rebuild makes sourcing resilient to any one actor dying: crawlerbros is now primary (free + richest fields), with the others as ordered fallbacks. Full bake-off & failover design in Scraper_Actor_Options.md.
TBD BizQuest as a cross-marketplace backstop (parseforge/bizquest-scraper) for when all BizBuySell actors are down at once. It lists largely the same businesses under its own businessId, so it needs a fuzzy name+city match (not ID dedupe) and a distinct BQ- id namespace before it can run alongside BBS.

Channel 2 · Outbound

Proprietary deal flow

Deals we go find. Instead of waiting for listings, we pick a thesis (an industry + geography we want to own), then build a verified list of business owners from public and paid data sources and reach out directly. No broker, no auction, no CIM up front — just owners who may not even be actively selling. This is run by the Proprietary Search pipeline (the build-prospect-list skill), a 20+ step discovery → enrichment → verification engine.

Live verticals: Tree Services · HVAC · Print Finishing · VoIP/UCaaS · Hazmat Geo: TX (OK, AR next) Target set: ~1,500 leads/run Tiering: TIER_1 → TIER_5 / NOT_READY
1Set the thesis (industry × geography)Manual

Pick the target pair to hunt — e.g., "tree services in TX." Theses are documented and drive everything downstream. The pipeline is config-driven, so a new vertical is just a new pair of config files.

Tools
Proprietary/Theses/ Industry_Taxonomy.md Market research (validation)
Inputs

  • Priority-state preference
Outputs

2Configure the target (YAML configs)Manual

Every pipeline script reads two config files: an industry YAML (NAICS codes, keyword regex, false-positive block/allow lists, certifications, name-extraction rules) and a geo YAML (state, metros, scrape cities, Secretary-of-State source, licensing sources). New industries are scaffolded from the closest existing template.

Tools

Inputs

  • Industry keyword regex

Outputs

3Discovery — cast the netManual

Pull candidate businesses from multiple public + scraped sources in parallel: federal registration/contract data, Google Maps, and the Better Business Bureau. Thesis-specific verticals add regulator pulls (e.g., EPA/TCEQ for hazmat). First paid step (Google Maps via Apify) is gated behind an explicit cost confirmation.

4Merge, dedupe & filter false positivesManual

Combine all sources into one set, drop duplicates and anything already in our tracker, then apply the config's block/allow lists to strip false positives (e.g., a "tree" match that's actually a dental office). A false-positive rate above 20% signals the config needs tuning.

Tools

Inputs

  • Existing lead history
  • Blocklist / allowlist regex
Outputs

  • FP + duplicate counts
5Enrichment — find the ownerManual

The highest-value work: attach a real owner name to each business. Secretary-of-State officer lookups are the most reliable source, backed by BBB detail scrapes, website scraping (owner names + certifications), state licensing cross-refs (TDLR), social profiles, and a Google-search backfill for stragglers. Owner names pass strict validation.

Outputs

Where both channels converge

Shared analysis & offer pipeline

Every deal — brokered or proprietary — that reaches a CIM or financials runs through the same core machine: score it, model it, decide, research the market, prep the conversation, capture the call, and make the offer. Three skills power the analysis — quick-lbo (BST + LBO), market-research (3-layer), and ioi (offer document) — alongside a self-hosted meeting-capture pipeline (step 7).

1CIM review — extract & normalizeShared

The quick-lbo skill parses the CIM PDF, pulls a multi-year P&L, and normalizes it — carefully avoiding double-counting add-backs (if the CIM already shows "Adjusted/Recast" figures, add-back rows are zeroed). Normalization logic is documented right in the model.

Runs: cim-deal-review — daily 8am / 1pm / 6pm, one unreviewed CIM per run (numeric + INB)
Tools
quick-lbo skill pdfplumber config.json (extracted financials)
Inputs
  • CIM PDF from CIMs/
  • Multi-year revenue / COGS / OpEx
  • Owner comp + add-backs
Outputs
  • Normalized P&L
  • Deal folder + _working/config.json
  • SDE composition breakdown
2BST — score business qualityShared

The Business Scoring Tool grades the business on six weighted categories (a weighted average, not a simple one). Location is intentionally low-weighted — a great business in PA shouldn't be penalized like a weak one in DFW.

Tools
populate_bst.py bst_template.xlsx
Inputs
  • CIM facts + normalized financials
  • Owner/management situation
  • Scoring config (1–10 per item)
Outputs
  • Weighted BST score /10
  • Category strengths / red flags
  • {id}_BST.xlsx
CategoryWeightWhat it measures
Size20%Deal economics — revenue, cash flow, margins, scalability
Product25%Defensibility / moat — differentiation, concentration, recurring revenue
Industry15%Market dynamics — fragmentation, growth, B2B stickiness
Owner / Mgmt25%Transition risk — tenure, motivation, team staying on
Growth10%Upside — clear expansion levers
Location5%Preference signal — within target markets (TX/OK/AR)
3LBO — model the deal economicsShared

A self-funded search LBO: full debt sizing, cash-flow waterfall, DSCR by year, and searcher/investor returns. Equity is 20% of total project cost — Thomas puts in $100K, investors fund the rest. Purchase price always uses the CIM's stated ask (low end), and every model records a price source.

Tools
populate_lbo.py lbo_template.xlsx Industry_Multiples.md (sanity)
Inputs
  • Normalized EBITDA / SDE
  • Purchase price + price source
  • Debt terms (SBA, seller note)
Outputs
  • {id}_LBO_Model.xlsx
  • DSCR, IRR, MOIC, FCF
  • Revenue-decline tolerance
Capital stack: SBA 7(a) + seller note + conventional + sponsor equity DSCR floor: 1.25× Target investor IRR: 20–35% Target MOIC: 3–5×

Reliability

Weekly review & self-audit

Two meta-tasks keep the engine honest. deal-flow-weekly-review (Sunday 7pm) produces the weekly quantitative + qualitative analysis — theses, missed opportunities, broker activity, multiple insights — refreshes the dashboard, and emails a summary. agent-audit-review (Tue/Fri 2am) audits the other tasks' runs for errors, retry loops, and token issues, then emails an improvement report. A running Journal.md logs every change, fix, and notable run.
